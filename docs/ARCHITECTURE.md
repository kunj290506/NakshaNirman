# Project Architecture

The CAD Floor Plan Generator is designed to bridge the gap between human requirements and complex architectural geometry. To achieve this, the application is divided into two distinct but deeply connected environments: a Python-driven backend that serves as the "brain", and a React-powered frontend that serves as the "canvas".

This document breaks down how these two sides operate together to deliver a seamless design experience.

## The Backend System (Intelligence and Geometry)

The backend is built on FastAPI and focuses entirely on processing logic, interfacing with AI models, and performing the heavy mathematical calculations required for floor plan generation. It is entirely stateless, meaning it relies on the frontend to provide the necessary data for each session.

### Key Components

*   **API Gateway (main.py):** This is the entry point for all frontend requests. It handles the initial payload, routing it to the appropriate internal services, and eventually returns the finished floor plan data back to the user.
*   **Input Validation (plan_validator.py):** Before any processing occurs, this module checks the user's requirements against physical and logical constraints. For example, it ensures that requested room sizes fit within the total plot area.
*   **Prompt Construction (prompt_builder.py):** Interfacing with Large Language Models requires specific, highly structured instructions. This component takes the validated user requirements and translates them into a dynamic prompt designed to yield precise spatial relationship data.
*   **AI Integration (llm.py):** This script manages the direct communication with external providers like GROQ and GROK. It sends the constructed prompt and safely parses the incoming JSON response, extracting the raw room connections and hierarchy.
*   **The Layout Engine (layout_engine.py):** This is the core mathematical processor of the application. It takes the abstract spatial relationships provided by the AI and converts them into precise X and Y coordinates. It calculates wall thicknesses, door placements, and exact room dimensions based on the user's initial plot boundaries.
*   **The Quality Engine (quality_engine.py):** Because AI generation can sometimes yield physically impossible results, this engine acts as a safety net. It reviews the coordinates generated by the Layout Engine, checking for overlapping rooms or inaccessible areas, automatically triggering a regeneration sequence if the plan fails these checks.

## The Frontend System (Interface and Visualization)

The frontend is built using React and Vite. Its primary responsibility is to provide an intuitive interface for capturing user constraints and to render the complex coordinate data returned by the backend into a cohesive visual floor plan.

### Key Components

*   **State Management (store/):** The frontend relies on a robust state management system to keep track of the user's current project settings, plot dimensions, and the resulting layout data. This prevents unnecessary refetching and ensures the interface feels highly responsive.
*   **Input Interface (components/RequirementsForm.jsx):** This module presents the user with a clean, structured form to define plot size, required rooms, styling preferences, and export requirements.
*   **The Rendering Canvas (components/canvas/):** Once data is returned from the backend, this highly specialized component takes over. It interprets the raw X and Y coordinates and draws the physical lines, walls, labels, and doors onto an HTML canvas, providing both 2D and 3D preview capabilities.
*   **Network Communications (services/api.js):** This isolates all the logic for communicating with the backend. It tracks loading states, handles potential network errors, and ensures the data flows cleanly into the application's global state.

## The Data Lifecycle

Understanding how data moves through the application provides the best overview of its functionality. Here is the step-by-step lifecycle of a single floor plan generation request:

1.  **Requirement Capture:** The user interacts with the frontend interface to define their plot boundaries and the required rooms. Let's say, a 50x50 plot with two bedrooms and one kitchen.
2.  **Data Transmission:** The frontend packages these constraints into a JSON payload and sends a POST request to the backend API.
3.  **Validation and Translation:** The backend validates that two bedrooms and a kitchen can physically fit into a 50x50 plot. It then builds a complex prompt describing these parameters.
4.  **AI Processing:** The backend queries the LLM, asking for an optimal topological arrangement for the requested rooms.
5.  **Coordinate Mapping:** The backend's Layout Engine receives the topology from the AI and maps it to physical, scaled X and Y coordinates on the requested plot size.
6.  **Quality Assurance:** The Quality Engine reviews the coordinates to ensure the generated plan is mathematically sound and that all rooms are accessible.
7.  **Final Delivery:** A completed JSON file containing exact wall definitions, room coordinates, and labels is returned to the frontend.
8.  **Visualization:** The frontend parses the coordinates and renders the final floor plan onto the viewing canvas for the user to explore and export.
